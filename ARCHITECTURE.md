# Architecture

## Компоненты
```
Источники данных → Коннекторы → Каталог → Поиск → MCP API → UI/Агент
```

## Слои

### 1. Коннекторы (`backend/connectors/`)
Читают источники данных и возвращают унифицированный формат `TableMeta`.

- `base.py` — абстрактный класс `BaseConnector` с интерфейсом `list_tables()` и `get_table_meta()`
- `sqlite.py` — читает SQLite через `sqlite3`
- `csv.py` — читает CSV через `pandas`

Добавить новый источник = написать новый файл коннектора. Остальное не трогать.

### 2. Каталог (`backend/catalog.py`)
Вызывает все коннекторы, собирает метаданные и сохраняет в `catalog.json`.

- `build_catalog()` — индексирует все источники
- `load_catalog()` — загружает индекс в память

### 3. Поиск (`backend/search.py`)
Принимает строку запроса, ищет по каталогу, возвращает ранжированные результаты.

- Keyword search — точное вхождение в имена таблиц, колонок, sample values
- Fuzzy search — `SequenceMatcher` для поиска с опечатками
- Scoring — результаты ранжируются по score от 0 до 1

### 4. Sensitivity (`backend/sensitivity.py`)
Автоматически помечает колонки с чувствительными данными (email, age, income, name и др.)

### 5. MCP сервер (`backend/mcp_server.py`)
FastAPI приложение с 4 эндпоинтами. Используется и UI и AI агентом.

### 6. UI (`frontend/`)
Простой HTML/JS интерфейс. Делает fetch запросы к MCP серверу.

## Поток данных
```
1. Старт → build_catalog() → catalog.json
2. Запрос → search(query) → load_catalog() → список SearchResult
3. Клик на результат → getSchema(source_id, table) → схема таблицы
```

## Ключевые решения

**Почему JSON каталог а не БД?**
Простота setup — не нужен отдельный сервер. Для текущего масштаба достаточно.

**Почему абстрактный BaseConnector?**
Расширяемость — новый коннектор пишется независимо от остальной системы.

**Почему keyword + fuzzy а не эмбеддинги?**
Быстрый старт без зависимостей от ML моделей. Эмбеддинги — следующий шаг.
